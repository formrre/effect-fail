\section{Introduction}

Algebraic effects provide a modular and lower overhead construction to flexibly add new operations to a monad comparatively to a monad transformer.
In transformer frameworks we need a new type class instance for every combination of effects and this fixes the implementation.
These provide ways to write programs causing abstract effect, which can be given concrete implementation afterwards using handlers. \highlight{We also want to provide a way to match on
patterns and fail in a default way, if they are not
satisfied}\dnoteM{This is the absolute key sentence I think, but it
needs more explanation. Why do we want to provide a way to match on
patterns? What exactly are patterns? And why might they fail?}.
Patterns play a crucial role in semantics definition of intermediate languages, such as CakeML compiler intermediate representations.
After some point in compiler we forget the information about typing and make certain operations just fail, if provided with wrong inputs.
One example from the CakeML compiler being the following,
%
\begin{lstlisting}
      | (El,[RefPtr ptr; Number i]) =>
        (case FLOOKUP s.refs ptr of
         | SOME (ValueArray xs) =>
            (if 0 <= i /\ i < & (LENGTH xs)
             then Rval (EL (Num i) xs, s)
             else Error)
         | _ => Error)
\end{lstlisting}
%

Here we lookup a value of pointer ptr and require it contains a specific kind of an array, if any other kind of an array is being pointed to we fail in a default way.
Same happens, when we get an out-of-bounds error, in the if case.
We would like to rewrite this using Haskell-style do notation as follows,
%
\begin{lstlisting}
	| (El,[RefPtr ptr; Number i]) => do
  	  (ValueArray xs) <- FLOOKUP s.refs ptr
	  cond (0 <= i /\ i < & (LENGTH xs))
	  return (Rval (EL (Num i) xs , s))
\end{lstlisting}
%

In Haskell this pattern matching bind provides a way to fail with a default value, when pattern matching on the result fails. Whereas normal do-notation,
%
\begin{haskell}
     do
	x <- action1
	action2
\end{haskell}
%
desugars to the following,
%
\begin{haskell}
    action1 >>= (\ x -> action2)
\end{haskell}
%

This requires, that action1 and action2 have correct monadic types.

The pattern matching bind, where pat is not a variable but a pattern with possible variables included is an extension of this idea,
%
\begin{haskell}
     do
	pat <- action1
	action2
\end{haskell}
%
desugars into,
%
\begin{haskell}
    action1 >>= (\ x -> case x of
	             pat -> action2
		     _ -> mfail)
\end{haskell}
%
Here x is selected as not to conflict with any user-defined variables. Not only does this desugaring require monadic bind, but also a new operation mfail.
This is captured in the Haskell type class MonadFail, which add this operation.
In haskell it is assumed due to the expected semantics of pattern matching that mfail satisfies the following law, for all x,
%
\begin{haskell}
	mfail >>= x = mfail
\end{haskell}
%

Standard formulations of effect monads can not support the interface
of MonadFail\dnoteM{First mention of this-- needs introducing first} as they are, because they use a Free Monad construction, which gives the simplest construction for which Monad laws hold.
We want not only to use Effect API\dnoteM{What is this?} for Monads but also give semantics of pattern matching.
As such we set out to find a monad supporting both failure and the Effect API.

Here we identify usefulness of MonadFail for reducing pattern matching
boilerplate\dnoteM{still yet to explain when/whyy this occurs} and provide a formalization of results combining Effect monad with Maybe transformer to provide the correct API.
In section 2 we discuss our construction of effect monads and prove that it is a monad and a graded monad.
In section 3 we provide the implementation of Effect API for this monad.
In section 4 we construct EffectFail and provide proofs that it is a MonadFail,
give instance lifting graded structure of Effect and provide the Effect API for this monad.
In section 5 we discuss future directions and applications we have in mind for this formalization.

When pattern matching can fail within do-notation in Haskell,

it desugars it into the following,
%
\begin{haskell}
  f >>= (\ x -> case x of
    pat -> remainingComputation
    _   -> mfail errorMsg)
\end{haskell}
%
The type of \haskin{mfail} is $String \rightarrow m\:a$ for some monad
$m$. We are interested in formulation not passing error
messages\dnoteM{What does this mean?} as these are ignored by most interesting instances of this.

The \haskin{MonadFail} type class\dnoteM{Where does it come from?}
 provides the \haskin{mfail} operation:
%
\begin{haskell}
class Monad m => MonadFail m where
    mfail : m a
\end{haskell}
\dnote{Citations to its use in the literature and discuss a bit what
  has been said about it there.}
Instances of the \haskin{MonadFail} class should satisfy
an additional axiom to the standard monad laws, for all $f : \texttt{a
  -> m b}$ then:
\begin{haskell}
	mfail >>= f == mfail
\end{haskell}
%
i.e., once a failure has occured, as given by \haskin{mfail}, then the
entire computation fails--subsequent sequentially composed computing
(e.g., \haskin{f} above) are ignored.

Thus, from the view of monads as monoids (in the category of
endofunctors), \haskin{mfail} identifies an element of the monoid
which is \emph{cancellative} or \emph{annihilating} with respect
to sequential composition; local failure becomes
global failure.

\subsection{Our construction in Agda}

We are working in dependently typed setting without type classes and
as such we need to do multiple tweaks. Without some notion of
positive and negative occurences,
we have to do the instantiation above in a single step.
Also Agda does not provide powerful enough type class resolution
mechanism and this would be unsatisfyingly metatheoretic. We construct
our effect monad by tweaking the Haskell construction slightly and
instead of providing an index to type-level list, we take a proof of
contained effect's membership in this list. As proofs of membership in
Agda are proof-relevant this at the same time acts as an index.
\begin{haskell}
	data Effect (E : List (Set -> Set)) (A : Set) : Set1 where
	  Pure : A -> Effect E A
          Impure : {F : Set} F X -> (F in E) -> (X -> Effect E A) -> Effect E A
\end{haskell}
\dnote{Explain the construction, also change to agda env}
Our construction changes the typeclass resolution constraint of Effect in Haskell to a proof that F occurs in the effect list.
We further refine this to provide names to every effect in the list
and thus being able to refer to each effect by a unique name, instead
of by its type.

\subsection{Algebraic effects as graded monads}

The concept of \emph{graded monads} generalises the notion of monads
from a single endofunctor (e.g., used to model a class of effects) to
an indexed family of endofunctors, essentially stratifying a model of
effects into different sub-models distinguished by the
index~\cite{DBLP:conf/popl/Katsumata14,DBLP:journals/corr/OrchardPM14}. The
indices have the structure of a monoid or pre-ordered monoid, which is
leveraged in the graded monad operations which are analogues of the
usual monad operations (and axioms).

\section{Maybe transformer gives an instance of MonadFail}

Now we focus on the next piece and that's giving the MonadFail instance to our Effect monad,
which justifies pattern matching failures
A standard approach of adding a notion of failure to an existing
monad is via a \emph{monad transformer}, typically
\haskin{MaybeT}~\cite{transformerpub}:
%
\begin{haskell}
newtype MaybeT m a = MaybeT { runMaybeT :: m (Maybe a) }
\end{haskell}
%
We prove that this transformer is a monad for any monad \haskin{m}.
We also prove that it is a lawful instance of \haskin{MonadFail} for
any monad \haskin{m}.
The first construction was introduced together with monad transformers
\cite{transformerpub}, the MonadFail instance in various Haskell
implementations \cite{mtl}. We give these constructions in Agda and
verify the laws of \haskin{Monad} and \haskin{MonadFail} given that m is a \haskin{Monad}.

\subsection{Effect API through Maybe transformer}

We lift the Effect API through Maybe transformer by identifying

\section{Future directions}
As much as we have implemented an effect API we would like to also reason about it.
One possible refinement is for handlers to not only take the instances of a given effect, but also quotient them to enforce that algebraic laws hold \cite{algtheorieshit}. This would force us to provide correctness proofs whenever mapping out of a particular effect and ensure that laws used to reason locally are enforced by handlers.
